# Codesurface Roadmap: Styling & Framework Expansion

## Strategic Position

Onlook, Bolt.new, Lovable, V0 = cloud-first AI app builders, all locked to Next.js + Tailwind.
Codesurface = local CLI tool for developers editing styles on their own running app.

**Our moat: multi-styling-system, multi-framework visual style editing that writes back to source.**

### Why breadth wins despite Tailwind's dominance

The numbers: Tailwind has 37% of developers, 75M+ monthly npm downloads. Next.js dominates meta-frameworks. AI tools (v0, Lovable, Bolt) all output Next.js + Tailwind + shadcn/ui. It's tempting to go deep on that stack.

But that market has 5+ funded competitors racing to own it:
- Onlook (YC-backed, building exactly this)
- v0 (Vercel — will add visual editing to their own output)
- Lovable, Bolt.new (will add visual refinement features)
- Cursor/Windsurf (adding visual preview)

Going deeper on Next.js + Tailwind means competing head-on with teams that have more resources and platform advantages.

Meanwhile, the developer working on a production app with CSS Modules, or CSS variables, or Bootstrap, or Vue, or plain Vite + React — they have **zero tools** for visual style editing that writes back to source. That's most of the industry. The 63% of developers NOT using Tailwind. The 30% NOT using React. Every existing codebase that wasn't generated by AI last week.

**Every styling system we add multiplies addressable market with no new competition. Every framework plugin opens an entirely unserved audience.**

## The Critical Gap

The `write-element` endpoint's `changes[]` path only converts CSS -> Tailwind classes. For css-variables, plain-css, and bootstrap projects, element-level style changes from the property panel have no write path. We detect 6 styling systems but can only write element styles for 1.

## Roadmap

### Phase 1: Per-element style editing for all styling systems

**Goal:** The property panel writes changes for ANY detected styling system, not just Tailwind.

#### 1a. Plain CSS write adapter
- **Effort:** Small
- **What:** Write CSS property/value directly to a stylesheet rule or inline style attribute
- **How:** Find the element via `data-source`, resolve its className to a CSS rule in the project's stylesheets, write the property/value. Fallback: write as inline `style` attribute on the JSX element (AST edit)
- **Files:** New `write-css.ts` adapter, update `write-element.ts` to dispatch by styling system type

#### 1b. CSS Variables element adapter
- **Effort:** Small
- **What:** Extend the existing token write path to work from the property panel's per-element flow
- **How:** When a property's computed value resolves to a CSS variable, write changes to the `--var` declaration. Already partially built — token editor writes CSS custom properties, this extends that to the element editing flow

#### 1c. CSS Modules support
- **Effort:** Small-Medium
- **Market:** The most common non-Tailwind styling approach in Next.js and Vite projects. Often used alongside Tailwind. #1 in State of CSS for non-utility approaches
- **How:** Detect `.module.css` imports via `import styles from '*.module.css'` patterns. Resolve `styles.foo` to the `.foo` rule in the module file. Write CSS property/value changes directly to that rule. The scoping is handled by the build tool — we just edit the source file

#### 1d. Bootstrap write adapter
- **Effort:** Medium
- **Market:** 22% of developers. Huge in enterprise/legacy, but those teams are slower to adopt new tools
- **How:** Map CSS property/value to Bootstrap utility class (`p-3`, `text-primary`) or SCSS variable override. Presets already exist in `presets/bootstrap.ts` with SCSS and CSS var override scanners

### Phase 2: Framework plugins

**Goal:** Codesurface works beyond Next.js.

#### 2a. Vite plugin for React
- **Effort:** Medium
- **Market:** Vite is the standard non-Next.js React setup. CRA is dead. This one plugin also covers Remix (Vite-based since v2) and any Vite + React app
- **How:**
  - Port the Babel visitor from `next-plugin/src/loader.ts` into a Vite `transform()` hook — the visitor is ~60 lines and framework-agnostic
  - Mount injection: detect `main.tsx` or `App.tsx` (configurable) and inject `<CodeSurface />`
  - Package as `@designtools/vite-plugin`
- **Complexity:** Entry point detection is less predictable than Next.js's `app/layout.tsx`. Provide a config option with smart defaults

#### 2b. Vue/Nuxt plugin
- **Effort:** Large
- **Market:** Vue has ~10% npm share but ~30% mindshare globally (huge in Asia, Europe, enterprise). Nuxt is its Next.js equivalent. **Zero visual editing tools exist for Vue.** This is a completely uncontested market
- **How:**
  - Source annotation: `@vue/compiler-sfc` to inject `data-source` into template elements
  - Mount injection: add component to app layout (Nuxt has predictable `app.vue`)
  - Style editing: Vue's `<style scoped>` maps directly to the plain CSS adapter from Phase 1

#### 2c. Astro plugin
- **Effort:** Medium (requires 2a as base)
- **Market:** Growing fast for content/marketing sites. Vite-based, has JSX support via React/Preact islands
- **How:** Astro integration wrapping the Vite plugin from 2a, with Astro-specific entry point detection (`src/layouts/`)

#### 2d. Svelte/SvelteKit plugin
- **Effort:** Large
- **Market:** Small but passionate community, zero competition in tooling
- **How:** Svelte preprocessor for `data-source` injection. `<style>` blocks map to the CSS adapter from Phase 1

### Phase 3: Token and design system depth

**Goal:** Best-in-class design token management across any project.

#### 3a. Tailwind v3 theme resolution
- **Effort:** Medium
- **What:** `tailwind-map.ts` currently uses hardcoded default scale values. Projects with custom themes get wrong class suggestions. Read and resolve `tailwind.config.js` (already detected but unused)

#### 3b. W3C Design Tokens import/export
- **Effort:** Medium
- **What:** Read `.tokens.json` files into the token map. Write changes back to JSON. Import/export between CSS variables and W3C format. Partially built in `presets/w3c-design-tokens.ts`

#### 3c. Sass/SCSS variable editing
- **Effort:** Medium
- **What:** Scan `$variable` declarations, display in token editor, write back to `.scss` files. Shadow writes already handle SCSS — extend to all token types. Declining relevance as native CSS replaces Sass features

## Recommended First Sprint

1. **Plain CSS write adapter** (1a) — gets multi-system element editing working
2. **CSS Variables element adapter** (1b) — extends existing token writes to property panel
3. **CSS Modules support** (1c) — covers the most common non-Tailwind pattern in the React ecosystem

These three make the pitch real: "codesurface works on your existing project, whatever styling approach you use."

Then: **Vite plugin** (2a) to break out of Next.js, followed by **Vue/Nuxt** (2b) to enter a completely uncontested market.

## What We're NOT Building

- Drag-and-drop element insertion/removal — too complex on real apps with data fetching, server components, conditional rendering. Easy on marketing sites but that's Webflow/Framer territory
- AI coding agent — different product, Onlook/Cursor territory
- Cloud sandboxes — local-first is our advantage, not a limitation
- Figma-style drawing tools — we edit existing apps, not create from scratch
- Deeper Next.js + Tailwind + shadcn features — that's the fight we lose against funded competitors with platform advantages
